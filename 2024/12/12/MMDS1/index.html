Mining of Massive Datasets

Frequent Item Sets

Definition

Given a support threshold s, sets of items that appear in at least s baskets.

Support for itemset I

Number of baskets containing all items in I

Association rules

$i_1,i_2,\cdots,i_k \rightarrow j$ : if a basket contains all of $i_1,\cdots,i_k$, it is likely to contain $j$

Confidence

$conf(I \rightarrow j)=\frac{support(I \cup j)}{support(I)}$

Interest

$Interest(I\rightarrow j)=conf(I \rightarrow j)- Pr[j]$

Positive: high conf low Pr, j is rare but it frequently co-occurs with I (complementary)

Negative: low conf high Pr, j is frequent but seldom co-occurs with I (substitutive)

Find Association Rules

To reduce number of rules

  • Maxmial frequent itemsets
    • No superset is frequent
  • Closed itemsets
    • No superset has the same count

Find Frequent Itemsets

Find Frequent Pairs

Naive Algo

For each basket of n items, generate its $\frac{n(n - 1)}{2}$pairs by two nested loops

Critical Resource

Main-Memory

Counting Pairs in Memory

  1. Approach 1
    1. Count all pairs using a matrix
    2. 4 bytes per pair
    3. pair{i, j} is at $(i - 1)(n - \frac{i}{2}) + j - 1$
    4. Total Number of Pairs: $\frac{n(n-1)}{2}$; Total bytes: $2n(n-1)$
    5. Triangular Matrix
  2. Approach 2
    1. Table of Triples[i, j, count(i, j)]
    2. 12 bytes per pair(but only for pairs with count > 0)
    3. Beats approach 1 if less than $\frac{1}{3}$ possible pairs actually occur

A-Priori Algorithm

Computation Model

  • Association-rule algo reads the data in passes(all baskets read in turn)
  • May read the data for multi passes
  • True cost usually the number of disk I/Os

A-Priori (A two-pass approach)

Key idea: monotonicity

if a set of item I appears at least s times, so does every subset J of I

Contrapositive for pairs

  • if item i doesn’t appear in s baskets, then no pair including i can appear in s baskets

Pass 1

Read baskets and count in main memory the occurrences of each individual item

  • Require only memory proportional to items

Pass 2

Read baskets again and count in main memory only those pairs where both element are frequent (from Pass 1)

  • Require memory proportional to $O(frequent-item^2)$
  • Plus a list of the frequent items

Frequent Triples

For each k, construct two sets of k-tuples

$C_k = candidate\text{ } k-tuples$ (support $\geq$ s)

  • Info from the k-1 pass

$L_k = \text{set of truly frequent k-tuples}$

Summary

  • One pass for each itemset size k
  • Needs room in main memory to count each candidate k-tuple

PCY(Park-Chen-Yu) Algo

  • Most memory is idle in pass 1 of A-Priori, Only store individual item counts

Pass 1

Maintain a hash table with as many buckets as fit in memory;

Keep a count for each bucket into which pairs of items are hashed

  • Each bucket just keep the count, not the actual pairs

If a bucket contains a frequent pair, then the bucket is surely frequent

None of the pairs for a bucket with total count less than s can be frequent

Pass 2

Only count pairs that hash to frequent buckets

To reserve memory

  1. Replace the buckets by a bit-vector
  2. Replace 4-byte integer by bits → $\frac{1}{32}$memory
  3. Decide which items are frequent and list them for pass 2
  • Count all pairs{i, j} that is candidate pair
    • Both i and j are frequent items
    • Pair{i, j} hashes to a bucket whose bit in the bit vector is 1(i.e a frequent bucket)
    • ⬆️ Necessary conditions for the pair to be frequent

Multistage Algo

Pass 1 of PCY, rehash only those pairs that qualify for pass 2 of PCY

  • i and j are frequent
  • {i, j} hashes to a frequent bucket from pass 1
  • Filter more infrequent pairs in the middle pass

Pass 3

Count {i, j} that

  • i and j are frequent items
  • Using the first hash function, the pair hashes to

a bucket whose bit in the first bit-vector is 1

  • Using the second hash function, the pair hashes to

a bucket whose bit in the second bit-vector is 1

Hints

  • Two hash functions have to be independent
  • check both hashes on pass 3

Multihash

Use several independent hash tables for pass 1

Risk: The average count doubles cuz half bucket numbers

  • Ensure most buckets wont reach count s

Less Pass

May miss some frequent Itemsets

Random Sampling

Run A-priori in main memory

No need to pay for disk I/O

Reduce Support threshold proportionally to match the sample size

SON Algo

Repeatedly read small subsets of the baskets into main memory and run an in-memory algo to find all frequent item sets

  • Not sampling but processing the entire file in memory-sized chunks

An itemset becomes a candidate if it is found to be frequent in any one or more subsets of the baskets

Pass 2

Count all the candidate itemsets and determine which are frequent in the entire set

Assumption: an itemset cannot be frequent in the entire set of baskets unless it is frequent in at least one subset

May miss some itemsets that are frequent in entire set but not in any subset

Distributed Version

Baskets distributed among many nodes

  • Compute frequent itemsets at each node
  • Distribute candidates to all nodes
  • Accumulate the counts of all candidates

Map-Reduce

Distributed File System

Chunk servers

File split into contiguous chunks

16-64MB each

each chunk replicated

try to keep replicas in different racks

Master node

Stores metadata about where files are stored

might be replicated

Client library for file access

Talks to master to find chunk servers

Connects directly to chunk servers to access data

Map-Reduce Diagram

Map Extract

Reduce Aggregate, summarise, filter or transform

Data Flow

Input and final output → global DFS

Intermediate results → Local FS

Output → another MapReduce task

Coordination:Master

  1. Take care of coordination
    1. Task status (idle, in-progress, completed)
    2. idle tasks get scheduled as workers become available
    3. When a map task completes, it sends the master the location and sizes of its R intermediate files
    4. Master pushes this info to reducers
  2. Ping workers periodically to detect failures

Failures

Map worker failure

  • Map tasks completed or in-progress at worker are reset to idle
  • Reschedule the task on another map worker
  • Reduce workers are notified when task is rescheduled on another worker

Reduce worker failure

  • Only in-progress tasks are reset to idle since reducer outputs are stored in global file system.
  • Reduce task is restarted

Master failure

  • Abort MapReduce task
  • Notify the client

Map and Reduce jobs

  1. M map tasks, R reduce tasks
  2. Make M $≥$ number of nodes in the cluster
  3. One dfs chunk per map
  4. Improve dynamic load balancing
  5. Speed up recovery from worker failures
  6. R is usually smaller than M

Refinements

Backup Tasks

Produce multiple backup copies of tasks near end of phase to save the effort of rerun a task due to machine failure

Combiners

Pre-aggregating values in the mapper to save network time

preq: reduce function is commutative and associative

Partition Function

Provide a custom hash function

Locality Sensitive Hashing

Problem: Find pairs of data points within distance threshold

[near-neighbours in high-dimensional space]

Jaccard similarity:

$sim(C_1,C_2)=\frac{|C_1\cap C_2|}{|C_1 \cup C_2|}$

Jaccard distance

$d(C_1,C_2)=1-\frac{|C_1 \cap C_2|}{|C_1 \cup C_2|}$

Similar Docs

Find near duplicate documents pairs

  1. Shingling: Convert documents to sets
  2. Min-hashing: Convert large sets to short signatures, while preserving similarity
  3. LSH: Focus on pairs of signatures likely to be from similar documents

1.Shingling

k-shingle/k-gram: a sequence of k tokens that appears in the doc

2. Min Hashing

Sets to boolean Matrices

Rows: elements(shingles)

Columns: sets(documents)

Find Similar columns

  1. Signatures of columns: Columns summary
  2. Examine pairs of signatures
  3. (Optional)Check the columns with similar signatures are really similar

Hashing Columns(Signatures)

Key: Hash each column C to a small signature h(c) such that

  1. h(c) small enough → signature fits in RAM
  2. $sim(C_1,C_2)$ same as the similarity of signatures $h(C_1)$ and $h(C_2)$

Goal Find a hash function h() such that

  1. $sim(C_1, C_2)$ high → high prob, $h(C_1)=h(C_2)$
  2. $sim(C_1, C_2)$ low →high prob, $h(C_1) \neq h(C_2)$

Min-Hashing

rows: random permutation $\pi$

hash func: $h_{\pi}(C) = min_{\pi} \pi(C)$

index of the first row in which column C has value 1

$Pr[h_{\pi}(C_1) = h_{\pi}(C_2)] = sim(C_1, C_2)$

3.LSH

Find docs with Jaccard similarity at least s

LSH for Min-Hash: Hash columns of signature matrix M several times

Candidate pairs: Hash to the same bucket

Partition M into Bands

  1. Matrix M → b bands of r rows
  2. Hash each band portion of each column to a hash table with k buckets
  3. Candidate column pairs: Same bucket where $band \geq 1$
  4. Tune b and r

Analysis

  1. Columns $C_1$ and $C_2$ have similarity $t$
  2. pick any band(r rows)
    1. Pr(all rows in band equal)= $t^r$
    2. Pr(some row in band unequal) = $1-t^r$
  3. Pr(No band identical) = $(1-t^r)^b$
  4. Pr(At least 1 band identical) = $1-(1-t^r)^b$

Tune b and r

$r*, b* = \min\limits_{r,b} (FP+FN) = \min\limits_{r,b} \int_{0}^s 1-(1-t^r)^bdt + \int_{s}^1(1-t^r)^bdt$

Summary

  • Adjust M, b, and r to match most pairs with similar signatures.
  • Filter out pairs with dissimilar signatures.
  • Verify candidate pairs’ similarity in main memory.
  • (Optional) In a 2nd pass, confirm that remaining pairs represent similar documents.

Shingling: Convert documents to sets

Min-Hashing: Convert large sets to short signatures, preserving similarity

Locality-Sensitive Hashing: Find candidate pairs of $sim\geq s$
